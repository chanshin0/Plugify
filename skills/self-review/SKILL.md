# self-review

## 목적

직전 어시스턴트 답변(또는 지정 대상)을 3개 다른 시각으로 검토해 **fake/over-engineering/사실 오류** 를 걸러낸다. 같은 모델이 같은 시각으로 3번 반복하면 자기 일관성 편향으로 첫 결정을 정당화하는 방향으로 수렴하기 때문에, 라운드별 루브릭을 **강제로 다르게** 한다.

## 트리거

수동 호출만:

| 인자 | 검토 대상 |
|---|---|
| `/self-review` | 직전 어시스턴트 답변 전체 |
| `/self-review --diff` | `git diff` (uncommitted) 결과 |
| `/self-review --last-commit` | `HEAD` 커밋의 변경 |
| `/self-review <file_path>` | 지정 파일 내용 |
| `/self-review --target "<text>"` | 임의 텍스트 |

자동 트리거(Stop hook 등) 금지 — 비용·소음 너무 크다.

## 사전 점검

```bash
command -v codex   # 없으면 R3 스킵 + 사용자 안내
```

미설치 시: "codex 미설치 — R1·R2 만 진행합니다." 출력 후 계속.

## Workflow

### R1 — 사실/근거 검증 (Claude)

**루브릭**: 답변에 인용된 모든 구체 항목이 실제 존재하는가.

검증 절차:
1. 답변에서 추출:
   - 파일 경로 → Read 또는 `ls` 로 존재 확인
   - 함수·심볼·변수명 → `grep -r` 로 존재 확인
   - 줄 번호 → 해당 위치 Read 로 확인
   - 명령어·CLI 플래그 → `command -v` + `<cmd> --help | grep` 확인
   - 외부 URL → 사용자가 제공하지 않은 URL 이면 의심 표기
2. 각 항목에 PASS / FAIL / WARN 표기

**산출**: R1 결과 — hallucinated 항목 목록 (없으면 "0 hallucinated").

### R2 — 반대 입장 (Claude)

**루브릭**: "이 답변이 틀렸다면 어디서 틀렸을까?"

검토 관점:
- **빠진 케이스/엣지** — 답변이 다루지 않은 시나리오
- **가정의 약점** — "X 라고 가정했는데 실제로는?"
- **삭제 가능 항목** — over-engineering, fake signal, scoring-driven 추가물
- **SSOT/DRY 위반** — 기존에 있는데 새로 만들었는가
- **사용자 의도 거리** — 사용자가 실제 원한 것과 답변의 GAP

각 항목에 **반박 강도** (강/중/약) 와 **근거** 1-2줄.

**산출**: R2 결과 — 반박 포인트 목록.

### R3 — Codex 외부 리뷰

**루브릭**: 독립 시니어 엔지니어 시각.

실행:
```bash
# 검토 대상을 임시 파일로 저장
TARGET_FILE=$(mktemp)
cat > "$TARGET_FILE" <<'EOF'
[검토 대상 본문]
EOF

# 컨텍스트 + 검토 대상 + 지시문을 한 프롬프트로
codex exec --skip-git-repo-check "$(cat <<EOF
당신은 독립적인 시니어 엔지니어다. 아래 답변을 비판적으로 평가하라.

## 사용자 원 요청
[원 요청 본문]

## 검토 대상 답변
$(cat "$TARGET_FILE")

## 평가 항목 (간결하게, actionable 만)
1. **사실 오류** — 잘못된 사실/api/문법
2. **누락** — 답변이 빠뜨린 중요 고려사항
3. **더 나은 접근** — 동일 목표를 더 단순/안전하게 달성하는 방법
4. **불필요한 항목** — 답변에서 빼도 되는 부분

각 항목은 1-2줄. 칭찬·요약 금지. 문제 없으면 "없음" 명시.
EOF
)"

rm -f "$TARGET_FILE"
```

**참고**:
- `--skip-git-repo-check` 는 Codex 가 git repo 외부에서도 동작하도록.
- 모델 변경 필요 시 `-m <model>` 추가.
- 응답이 길면 stdout 마지막 N줄만 추출해도 됨.
- Codex 가 timeout/실패 → R3 결과 "실패 (사유)" 표기 후 계속.

**산출**: R3 결과 — Codex 응답 본문.

### Synthesis

3개 결과를 모아 **합집합이 아니라 actionable 만 필터**:

| 우선순위 | 정의 |
|---|---|
| **High** | 명백한 사실 오류·hallucination·심각한 누락. 즉시 수정 필요 |
| **Medium** | 개선 권장 (R2/R3 합의 또는 단일 라운드의 강한 근거) |
| **Low** | 스타일·취향·노이즈. 무시 권장 |

각 항목 옆에 **출처 라운드** 표기 (예: `[R1]`, `[R2&R3]`).

**중요**: R1/R2 와 R3 가 **합의** 한 항목 우선. 단일 라운드 단독 의견은 강한 근거 없으면 Medium 이하.

### 사용자 결정 게이트 (필수)

`AskUserQuestion` 으로 High/Medium 항목 제시:

> 위 항목 중 어느 것을 답변에 반영할까요?
> - High 전체 적용
> - 항목별 선택
> - 적용 안 함 (리뷰만 수행)

선택한 항목만 후속 수정. **자동 일괄 수정 금지** — 사용자가 fake 를 거를 마지막 게이트.

## 출력 포맷

```
## Self-Review: <대상 요약>

### R1 — 사실 검증 (Claude)
- ✅ <건수>건 PASS
- ❌ <항목> — <사유>

### R2 — 반대 입장 (Claude)
- [강] <반박 1> — <근거>
- [중] <반박 2> — <근거>

### R3 — Codex 외부 리뷰
[Codex 응답 본문 또는 요약]

### Synthesis
- [High] <항목> [R1, R3]
- [Medium] <항목> [R2&R3]
- [Low] <항목> [R2]

### 다음 단계
AskUserQuestion 으로 적용 항목 선택.
```

## 원칙

- **각 라운드 다른 루브릭** — 같은 시각 반복 금지.
- **자기 일관성 편향 인정** — R1/R2 만으로는 부족. R3 외부 모델이 핵심.
- **자동 적용 금지** — 사용자가 항목별로 거부할 수 있어야 한다 (지난 워크플로우에서 fake 4개 거른 게 이 게이트의 역할).
- **Synthesis 는 합집합이 아님** — 모든 지적을 반영하면 노이즈가 답변을 덮어쓴다. 합의된 항목만.
- **검토 대상 자체를 수정하지 않음** — 이 스킬은 검토 + 제안까지. 수정은 사용자 결정 후 별도 단계.

## 한계

- **같은 회사의 close-source 모델** (Claude·GPT 계열) 끼리는 학습 분포 겹쳐 독립성이 완벽하지 않음. 그래도 다른 모델/프롬프트라 R1/R2 단독보다는 효과적.
- **속도 비용** — 라운드당 30초~2분. 모든 답변에 적용 비추천. 중요한 결정·설계·문서 변경 시에만.
- **Codex 의 컨텍스트 한계** — 매우 큰 답변은 잘라서 넘기거나 핵심 부분만 발췌.
